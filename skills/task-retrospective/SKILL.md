# 任务复盘

仅在用户明确要求“复盘任务”“分析失败原因”“总结任务过程”“回顾执行结果”时使用本 skill。不要在任务失败后自动触发，也不要把普通进度查询当成复盘。

## 严格复盘模式

默认启用“严格复盘模式”。严格复盘的目标是回答“这个任务为什么失败/偏离、证据是什么、后续应该怎么做”，而不是替原任务继续调查、补齐证据、验证源码、重跑任务或直接进入修复。

严格复盘模式下，只允许依赖当前任务系统直接可见的事实来源：

- `task_summary`
- `task_list`
- `task_failed_nodes`
- `task_progress`
- `task_node_detail`

严格复盘模式下，不要主动执行这些额外操作：

- 不要主动搜索工作区源码、读取任意仓库文件、扫描目录、grep 日志、打开会话文件或运行命令。
- 不要把磁盘上的任务目录、快照文件、工件目录、日志目录当作默认补充信息源。
- 不要重跑任务、重试节点、验证修复、修改代码或补写报告。
- 不要因为“想把复盘做完整”就自行扩展到纯 retrospective 之外的调查。

如果某条结论只能靠额外调查确认，必须明确写成“基于当前任务可见信息无法确认”，而不是自行补查。

## 任务定位

优先按下面顺序定位目标任务：

1. 用户已提供 `task_id` 时，先调用 `task_failed_nodes` 获取失败节点，再调用 `task_progress` 查看整体树和任务状态。
2. 用户未提供 `task_id`，但表达了“最近失败任务”“某个后台任务”之类的范围时，先调用 `task_list` 缩小候选。
3. 用户想先看整体情况时，可先调用 `task_summary`，再决定是否继续调用 `task_list` 或 `task_progress`。

如果候选任务不唯一，先追问一次，并给出候选 `task_id` 与简短摘要。不要猜测用户指的是哪一个任务。

## 复盘流程

按下面流程执行：

1. 获取任务事实：先用 `task_failed_nodes` 获取失败节点 id，再用 `task_progress` 获取状态、树状结构、最新节点输出、任务摘要。
2. 先定位失败节点：结合 `task_progress` 返回的节点树，判断每个失败节点位于哪一层、属于哪条分支，以及是否是验收节点失败、子节点失败上浮、或根节点直接失败。
3. 逐个分析失败节点：对每个失败节点调用 `task_node_detail(task_id, 节点id)`，查看节点详情、`failure_reason`、`final_output`、`check_result`、`execution_trace` 以及关联工件；如果当前可见信息不足以确认，只能写成“基于节点树/节点详情可见信息推断”。
4. 顺着树往上梳理：沿父子关系向上追溯，说明局部失败如何传导为更高层偏差或最终任务失败。
5. 提炼可行动结论：给出后续建议，而不是只复述日志。

不要虚构完整日志、未展示的工件内容、隐藏上下文或未调用到的工具结果。

## 安全降级

当理想流程走不通时，按下面顺序安全降级，不要越级自行扩查：

1. **没有 `task_id` 或候选不唯一**：只用 `task_list` / `task_summary` 缩小候选；如仍不唯一，追问一次并停止。
2. **`task_failed_nodes` 不可用或结果不完整**：退回到 `task_progress`，只基于节点树和任务状态给出结构化判断。
3. **`task_node_detail` 不可用或信息不足**：只基于 `task_progress` 和已可见摘要输出“当前可见事实 + 未确认点”，不要补做仓库调查。
4. **节点详情提到了工件但没有正文/摘录**：明确记录“当前任务工具未暴露足够工件内容”，不要自行去磁盘路径、工件目录、快照文件中补读。
5. **仍无法确认根因**：停止在“可见信息级别”的结论层，明确写出信息缺口与推荐的下一步调查，而不是继续扩展调查。

降级后的结论必须显式标明“这是事实”还是“这是基于当前可见信息的推断”。

## 何时已经超出纯 retrospective

出现以下任一情况，都说明已经超出纯 retrospective 的范畴：

- 需要读取仓库源码、目录、日志、stdout/stderr、session 文件或快照文件来确认根因。
- 需要重新核对工件中的方法名、类名、配置键、命令名是否真实出现在源码中。
- 需要复现问题、重跑节点、验证某个修复方向、比较多版结果、补写缺失报告。
- 需要对多个失败分支做大范围深挖，而不仅是总结当前任务系统已暴露的事实。

一旦超出纯 retrospective，不要在本次复盘里继续手工扩查；要么停在严格复盘结论，要么转入“补充调查任务”。

## 超范围处理：异步任务或派生子节点

当用户明确要求继续深挖，或者当前宿主运行时明确要求继续推进时，按下面规则转交：

- **创建异步任务 (`create_async_task`)**：适用于跨多个失败分支、多文件、多工件、可能耗时较长，或结果会被当成单独交付物继续使用的调查。
- **派生子节点 (`spawn_child_nodes`)**：适用于你已经处在任务执行上下文里，且后续调查可以按失败分支、失败节点、证据主题拆成相对独立的小块并行推进。

转交时必须写清楚这些内容：

- 来源任务 `task_id`
- 目标失败节点 id 或失败分支范围
- 当前已经确认的事实
- 当前无法确认的问题
- 允许使用的证据来源
- 不允许的操作（例如“不修改代码”“不重跑原任务”）
- 期望产出（例如“补充调查结论”“严格证据校准报告”“修复建议草案”）

如果用户没有明确要求继续深挖，就不要自动创建异步任务或派生子节点。默认只给出“若要继续，建议这样转交”的下一步建议。

## 输出要求

默认按照 `references/output-template.md` 的结构输出，覆盖这些部分：

- 任务概况
- 结果判断
- 关键证据
- 失败原因或偏差原因
- 可复用经验
- 下一步建议

并额外满足这些要求：

- 第一部分明确写出本次是否按“严格复盘模式”执行。
- 明确列出本次复盘实际使用的事实来源边界。
- 每条关键证据尽量标注来源于哪个任务工具。
- 任何超出当前任务可见信息的部分，都必须标成“推断”或“待补充调查”。
- 如果信息不足，明确写出缺口，例如“基于当前任务摘要/进度可见信息，无法确认具体失败工具输出”。
