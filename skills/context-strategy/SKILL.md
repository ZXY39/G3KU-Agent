# Context Strategy

统一上下文策略：指导 G3KU 在长会话中管理系统化上下文，包括保存、恢复、预算控制、范围标注、裁剪压缩与会话连续性。与 `memory` skill 和 `submit_next_stage` 机制协作，防止上下文溢出和信息丢失。

## 何时触发

**触发**：
- 会话上下文接近容量限制（>60% 预警，>80% 临界）
- 执行高风险操作前需保存上下文状态（大规模重构、批量派生子节点、复杂工具链）
- 会话压缩/重启后需要恢复工作上下文
- 需要跨会话或跨阶段保持项目连续性
- 用户明确要求保存、恢复或检查上下文状态
- 检测到上下文膨胀（响应变慢、重复信息过多、token 成本上升）

**不触发**：
- 简单问答或单步操作
- 上下文使用率很低（<40%）且无复杂操作计划
- 一次性原型验证

## 核心工作流

上下文策略按 **PREVENT → PRESERVE → RECOVER → CONTINUE** 四个阶段运作：

### 1. PREVENT（预防）— 预算与裁剪

详见 `references/budget-and-trimming.md`。

- **分区管理**：将上下文分为固定锚点（项目核心信息）、可变上下文（当前任务状态）、探索缓存（临时推理结果）
- **阶梯式预警**：
  - 安全区（<60%）：正常运行
  - 预警区（60-80%）：主动裁剪探索缓存，压缩冗余推理
  - 临界区（>80%）：执行保存 + 请求压缩/阶段切换
  - 危险区（>90%）：强制检查点保存，停止非必需操作
- **裁剪优先级**（从最低价值到最高价值移除）：工具原始输出 > 探索性推理 > 中间讨论 > 已完成阶段摘要 > 核心任务目标

### 2. PRESERVE（保存）— 检查点与快照

详见 `references/checkpoint-and-save.md`。

- **自动触发条件**：阶段完成、高风险操作前、进入预警区后每步操作
- **检查点内容**：当前目标、已完成工作、关键决策、待完成项、活跃引用
- **保存格式**：结构化 YAML/Markdown，包含 timestamp、stage_id、context_version
- **与 stage 机制集成**：每次 `submit_next_stage` 的 `completed_stage_summary` 和 `key_refs` 即为正式检查点

### 3. RECOVER（恢复）— 压缩后重建

详见 `references/recovery.md`。

- **检测压缩**：发现上下文丢失关键信息时触发
- **恢复流程**：检查最近检查点 → 加载固定锚点 → 按恢复计划补回可变上下文 → 验证恢复完整性
- **渐进恢复**：先恢复核心目标和当前阶段，再按需补回细节

### 4. CONTINUE（延续）— 跨会话连续性

详见 `references/continuity.md`。

- **会话结束前**：更新 tracks 状态、记录 blockers 和决策、提交最终摘要
- **新会话启动**：读取最近检查点、验证上下文时效性、恢复活跃任务
- **与 memory skill 协作**：关键决策通过 `memory` 持久化，临时状态不写入长期记忆

## 范围标签（Scope Tags）

简化版范围标注约定，用于标记消息的上下文范围和记忆意图：

| 标签 | 含义 | 示例 |
|------|------|------|
| `[ISO: <topic>]` | 隔离上下文，不引入外部信息 | `[ISO: database-migration]` |
| `[SCOPE: <topic>]` | 限定到特定范围 | `[SCOPE: api-refactor]` |
| `[GLOBAL]` | 允许跨范围复用 | `[GLOBAL] 之前的错误处理方案可复用` |
| `[NOMEM]` | 不写入长期记忆 | `[NOMEM] 临时测试` |
| `[REM]` | 建议持久化（依赖 memory 后端） | `[REM] 项目使用 PostgreSQL` |

冲突处理：后出现的标签优先；`[ISO]` 与 `[GLOBAL]` 冲突时需询问用户。

默认行为（无标签时）：保守处理，不主动跨范围混用上下文。

## 与 G3KU 其他系统的协作

详见 `references/integration.md`。

- **与 stage 机制**：`submit_next_stage` 的 summary 和 key_refs 是正式检查点；阶段切换是上下文裁剪的最佳时机
- **与 memory skill**：持久决策写入 memory，临时状态留在会话上下文；memory 用于跨会话恢复的关键锚点
- **与 child nodes**：派生子节点前保存父节点上下文；子节点失败重派时继承父节点检查点
- **与 skill-creator**：修改上下文策略本身时，遵循 skill-creator 的更新工作流

## 禁止事项

- 不得在上下文 >80% 时开始新的大型任务
- 不得跳过检查点直接执行高风险操作
- 不得把临时探索结果写入长期记忆
- 不得假设 AI 跨会话记住任何未记录的信息
- 不得在恢复前就假设上下文完整
