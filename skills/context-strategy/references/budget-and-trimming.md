# 预算管理与上下文裁剪

## 上下文分区

将当前会话上下文分为三个层级：

### Layer 1: 固定锚点（不可裁剪）
- 核心任务目标（`goal`、`core_requirement`）
- 项目根目录和关键路径
- 已提交的阶段检查结果（`submit_next_stage` 的 summary）
- 关键决策记录（写入 memory 的持久信息）

### Layer 2: 可变上下文（选择性裁剪）
- 当前阶段目标和工作计划
- 活跃工具调用结果
- 子节点目标与状态
- 正在进行中的分析/设计决策

### Layer 3: 探索缓存（优先裁剪）
- 工具调用的原始输出全文
- 探索性推理过程和中间假设
- 候选方案的详细对比过程
- 试错过程中的失败记录（保留结论即可）

## 预算阈值与行动

| 使用率 | 区域 | 行动 |
|--------|------|------|
| <60% | 安全区 | 正常运行，无需干预 |
| 60-80% | 预警区 | 裁剪 Layer 3 内容；压缩冗长推理为结论 |
| 80-90% | 临界区 | 创建检查点；裁剪 Layer 3 + 部分 Layer 2；准备阶段切换 |
| >90% | 危险区 | 强制保存检查点；停止非必需操作；请求压缩或创建新阶段 |

## 裁剪策略

### 优先级（从低到高保留）
1. 工具调用的原始输出（保留摘要和关键数据即可）
2. 探索性推理过程（保留最终结论和决策依据）
3. 候选方案的对比细节（保留最终选择及理由）
4. 失败分支的调试过程（保留失败原因和解决方案）
5. 已完成阶段的详细过程（保留阶段摘要和关键引用）
6. 当前活跃阶段的全部细节（最高优先级保留）
7. 核心任务目标和约束（始终保留）

### 裁剪操作
- **压缩**：将冗长内容替换为一行摘要，保持信息链完整
- **归档**：将完整内容写入文件（如 `.g3ku/context-archive/`），上下文中只留引用
- **丢弃**：确认无后续用途的临时推理内容直接移除

## 预估上下文大小

由于无法直接获取 token 计数，使用以下经验指标：
- 对话历史中的消息数量和内容长度
- 工具调用次数和输出大小
- 是否已触发过平台自动压缩

经验法则：
- 超过 15 轮密集工具调用后，开始关注上下文状态
- `submit_next_stage` 是主动裁剪的最佳时机
- 每次大段内容输出（>200 行）后评估上下文状态

## 与 compaction-survival 候选的关系

借鉴了 compaction-surface 的 Write-Ahead Logging (WAL) 思想，但不采用其具体实现。
改为使用 G3KU 原生的 `submit_next_stage` 机制作为检查点载体，更贴合项目架构。
