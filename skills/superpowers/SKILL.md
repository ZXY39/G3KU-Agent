# Superpowers - AI 辅助软件开发技能库

## 概述

Superpowers 是一个面向 AI 辅助软件开发的开放式技能库（skill library），由 obra 创建并维护。该项目不是传统意义上的代码生成工具或脚手架，而是一套**结构化的开发工作流技能集合**，旨在将优秀开发者的实践经验编码为可重复、可验证的指令模式，供 AI 代理（Agent）和开发者协同执行。

项目地址：https://github.com/obra/superpowers

---

## 核心理念

### 1. AI 增强而非替代（AI-Augmented, Not AI-Replaced）

Superpowers 的核心哲学是：AI 不应取代开发者做决策，而是放大开发者的能力。开发者保持对架构、质量标准和业务逻辑的控制权，AI 则承担重复性执行、模式验证和文档生成等任务。

### 2. 技能驱动开发（Skill-Driven Development）

将优秀开发者的工作习惯编码为显式技能（skill），每个技能是一个结构化的 Markdown 文件，包含：
- 前置检查（prerequisites）
- 执行步骤（step-by-step instructions）
- 验证标准（validation criteria）
- 故障排除指南（troubleshooting）

### 3. AI 原生设计原则

- **精确胜于冗长**：指令简洁明确，避免歧义
- **结构化输出**：模板化格式确保一致性和可验证性
- **可组合性**：技能之间可串联协作，形成复合工作流
- **人类可读 + AI 可解析**：双重受众设计

### 4. 测试驱动与证据导向

强调以测试作为开发驱动力，所有开发活动都应有可验证的证据支撑，而非依赖主观判断。

## 典型工作流

### 工作流一：测试驱动开发（TDD）循环

```
明确需求 → 编写失败测试 → 运行确认失败
    → 实现最小代码 → 运行确认通过 → 重构并验证
```

### 工作流二：代码重构与审查

```
确定重构范围 → 创建安全网测试 → 执行重构
    → 运行测试验证 → 提交代码审查 → 合并
```

### 工作流三：AI 辅助任务分解

```
接收复杂任务 → 分解为子任务 → 分配执行顺序
    → 定义每个子任务的输入/输出 → 逐个子任务执行验证 → 整合结果
```

### 工作流四：文档生成与审计

```
扫描代码库 → 识别缺失/过时文档 → 生成文档草稿
    → 人工审查修正 → 提交更新 → 定期审计
```

## 能力与边界

### 能做什么

- **规范化开发流程**：将最佳实践转化为可执行的技能指令
- **AI 代理赋能**：为 LLM/Agent 提供结构化的工作模式，减少幻觉和偏离
- **知识沉淀**：将团队经验固化为可复用的技能资产
- **跨项目复用**：技能独立于具体项目，可在不同代码库间迁移
- **质量保证**：通过内置验证步骤确保每个阶段的输出质量

### 做不到什么

- **不替代开发者判断**：架构决策、业务逻辑判断仍需人类主导
- **不是代码生成器**：不会自动生成完整项目代码，而是指导开发过程
- **不处理运行时行为**：关注开发工作流，不涉及运行时监控或运维
- **不保证技术选型正确性**：技能提供方法论，不替代技术评估

## 适用与不适用场景

### 适用场景 ✅

- 团队引入或转型 AI 辅助开发，需要结构化工作模式
- 新项目启动，需要快速建立开发规范和工作流
- 代码库重构，需要系统化的质量保证流程
- 团队知识传承，将资深开发者经验编码为可复用资产
- 已有 AI Agent 框架，需要补充专业技能指令集

### 不适用场景 ❌

- 期望一键生成完整应用的场景
- 完全自动化无人值守的开发流程
- 运行时性能监控或生产环境运维
- 非软件开发领域（如数据分析、机器学习训练等）
- 团队拒绝 AI 辅助开发，坚持纯手工编码

## 输入输出预期

### 典型输入

- 开发任务描述（用户需求、缺陷报告、功能规格）
- 代码库访问权限（文件结构、源代码、测试用例）
- 配置信息（项目类型、技术栈、编码规范）
- 技能选择指令（根据任务类型选用对应技能）

### 典型输出

- 结构化的开发步骤执行结果
- 代码变更（带验证的修改）
- 文档更新（API 文档、README、架构说明）
- 测试报告与覆盖率分析
- 代码审查意见与改进建议

## 优势与局限

### 优势

1. **系统性**：覆盖软件开发生命周期的主要环节
2. **可验证**：每个技能都包含明确的验证标准
3. **可演进**：开放式技能库，社区可贡献新技能
4. **低侵入**：不强制改变现有技术栈，可渐进式引入
5. **透明度高**：所有技能以 Markdown 开放，易于审计和调整

### 局限

1. **学习曲线**：需要理解技能体系和工作流模式
2. **AI 依赖**：充分发挥价值需要配合 LLM/Agent 使用
3. **语言偏好**：技能文档以英文为主，中文团队需要本地化
4. **维护成本**：技能库需要随技术生态演进而持续更新
5. **情境适配**：通用技能可能需要针对具体项目裁剪

## 与一般开发助手的关键差异

| 维度 | Superpowers | 一般开发助手/脚手架 |
|------|-------------|-------------------|
| 核心定位 | 结构化工作流技能库 | 代码生成/模板填充 |
| 决策权归属 | 开发者保持控制 | 助手可能偏离需求 |
| 输出形式 | 分步骤、可验证的流程 | 一次性代码片段 |
| 质量保证 | 内置验证检查点 | 依赖开发者自行检查 |
| 复用方式 | 技能组合、可定制 | 模板套用、固定模式 |
| 知识沉淀 | 经验编码为技能资产 | 零散的建议 |
| AI 适配 | 专为 Agent 设计的结构化指令 | 通用对话式交互 |
