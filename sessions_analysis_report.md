### 2.2 前缀含义解析

#### 模式 A: `china_qqbot_default_dm`

| 字段段 | 含义 | 推测值 |
|--------|------|--------|
| `china` | 地区/市场定位 | 中国大陆 |

**解读**：这是一个 QQ 机器人的单例默认私信通道。文件唯一性暗示 QQ bot 仅有一个默认 DM 会话，可能因为 QQ 机器人以群聊或统一通道方式运作，不需要为每个用户创建独立会话文件。

#### 模式 B: `web_ceo-{hash}`

| 字段段 | 含义 | 推测值 |
|--------|------|--------|
| `web` | 平台/接入方式 | Web 端 |
| `ceo` | 人格（persona）/角色 | CEO 角色扮演的 AI 助手 |
| `{hash}` | 会话唯一标识符 | 12 位十六进制字符串 |

**解读**：Web 端的 CEO 人格会话，每个用户/浏览器会话生成独立的 hash 标识。hash 看起来是截断后的 UUID 或随机 hex（12 字符 = 48 bit），用于区分不同的 Web 用户会话。

### 2.3 Hash 用途分析

- **长度**: 12 位十六进制字符（48 bit）
- **生成方式**: 推测为 `uuid4().hex[:12]` 或 `secrets.token_hex(6)`
- **用途**: 会话唯一标识，关联到 Web 端的 cookie/session/token
- **碰撞概率**: 12 hex = 2^48 ≈ 281 万亿种组合，在会话量 < 10000 时碰撞概率接近于零
- **可追溯性**: 不可逆哈希，不泄露用户身份（隐私友好）

## 3. 存储载体分析

### 3.1 为什么选择 `.jsonl` 而非 `.json`

| 维度 | `.jsonl` | `.json` (单文件数组) |
|------|----------|----------------------|
| **追加写入** | O(1)，直接 `echo >> file` | O(n)，需读取整个数组追加后重写 |
| **流式读取** | 逐行解析，内存占用恒定 | 需加载整个 JSON 到内存 |
| **容错能力** | 单行损坏不影响其他行 | 一处语法错误整个文件不可解析 |
| **并发安全** | 行级追加天然原子（多数 OS） | 需锁定整个文件 |
| **增量备份** | 可按行切分/过滤 | 必须整体处理 |
| **日志特性** | 天然适合日志/追加语义 | 不适合增量写入场景 |

**结论**：`.jsonl` 是聊天会话的理想存储格式，因为：
1. **对话天然追加**：每条新消息只需追加一行到文件末尾
2. **长对话友好**：大会话可达 57KB，逐行解析避免内存问题
3. **崩溃安全**：进程异常退出时，已写入的行仍然可读
4. **流式输出**：AI 生成回复时可逐条写回用户消息，无需等待完整转储

### 3.2 JSONL 内部结构（抽样验证）

每条记录的典型结构：

```jsonl
{"type":"metadata","value":{"created_at":"2026-03-30T17:58:22.793079-07:00","updated_at":"2026-03-30T18:04:59.576727-07:00","commit_turn_counter":3,"_meta_schema_version":1}}
{"type":"message","value":{"role":"user","content":"...","timestamp":"2026-03-30T18:02:14.972697-07:00"}}
{"type":"message","value":{"role":"assistant","content":"...","timestamp":"2026-03-30T18:04:59.566723-07:00"}}
```

关键字段：
- `type`: `"metadata"` | `"message"` — 控制行类型
- `metadata.created_at`: 会话创建时间（ISO 8601）
- `metadata.updated_at`: 最后活跃时间
- `metadata.commit_turn_counter`: 交互轮次计数
- `message.role`: `"user"` | `"assistant"`
- `message.timestamp`: 消息时间戳

### 3.3 为什么不是数据库或其他格式

| 格式 | 优势 | 劣势 |
|------|------|------|
| SQLite | 查询能力强 | 需要维护数据库连接、迁移 |
| JSON（单文件） | 结构紧凑 | 追加写入代价大，并发不安全 |
| Redis | 高性能 | 需要运行外部服务，持久化管理复杂 |
| **JSONL（当前）** | 零依赖、追加安全、可读 | 复杂查询需要全文件扫描 |

**选择 JSONL 的原因**：G3KU 是轻量级框架，追求零外部依赖。JSONL 文件可以被任何文本工具读取，不需要额外的数据库服务，完美契合"会话历史按需读取"的场景。

## 6. 总结

### 关键发现

1. **完全扁平的目录结构**：无子目录，适合当前 25 个会话的规模。未来若超过 1000 个会话，建议按时间分目录。

2. **Web CEO 会话占绝对主流（96%）**：说明 G3KU 的主要使用方式是 Web 端与 AI CEO 人格交互。QQ 机器人可能是实验性或辅助渠道。

3. **大量短命会话（72% < 1KB）**：这是 Web AI 应用的典型特征——用户尝试性访问后快速离开。建议考虑会话自动清理/归档策略。

4. **JSONL 格式选择合理**：零依赖、追加安全、流式读取，完美匹配聊天会话的 I/O 模式。

5. **Hash 标识设计良好**：12 位 hex 提供了足够的唯一性空间，同时不泄露用户身份信息。
