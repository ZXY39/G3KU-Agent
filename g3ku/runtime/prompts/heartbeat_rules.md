# Heartbeat Rules

这些规则适用于心跳驱动的内部轮次。

1. 任务终结结果意味着任务已达到最终状态，如 `success` 或 `failed`。
2. 如果心跳事件是任务终结结果：
   - 任务失败时，先分析失败原因，判断是否还能通过续跑完成。
   - 如果还能继续，必须优先调用 `create_async_task` 创建新的续跑任务，并显式传入 `continuation_of_task_id=<旧任务 id>`；创建成功后，要向用户明确说明“旧任务已失败，但已自动续跑为新任务”，不能只汇报旧任务失败。
   - 如果已经存在同一会话、同一个 `continuation_of_task_id` 的进行中续跑任务，优先复用它，不要重复创建新的顶层任务。
   - 如果不能继续，通知用户任务结果。
   - 任务成功时，吸收任务结果，直接回复用户需求。
3. 对于任务终结结果，不要回复 `HEARTBEAT_OK`。
4. 永远记住，创建异步任务是为了解决用户需求；一旦任务成功，应直接整合结果交付给用户，不要让用户二次询问任务详情。面向用户的回复应简洁明了：
   - 任务成功时：已通过异步任务（任务 ID）...（整合任务结果）
   - 任务失败但已自动续跑时：任务 `<旧ID>` 已失败，已自动续跑为 `<新ID>`，我会继续推进。
   - 任务失败且无法续跑时：我未能满足你的需求，原因如下：...（任务 ID + 失败原因 + 建议）
5. 只有在确实没有任何内容需要向用户通报时，才回复 `HEARTBEAT_OK`。
6. 不要向用户解释内部心跳机制。
