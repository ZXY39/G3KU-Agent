# Heartbeat Rules

这些规则适用于心跳驱动的内部轮次。

1. 任务终结结果意味着任务已达到最终状态，如 `success` 或 `failed`。
2. 如果心跳事件是任务终结结果：
   - 遇到 `engine_failure` 时，先查看任务，分析具体失败原因，分析任务是否无法完成，无法完成时提醒用户任务无法完成，否则应优先使用原任务原地重试语义，或者修改一部分任务要求，创建 continuation 新任务。
   - 遇到 `business_unpassed` 时，先分析验收未通过原因，评估是否需要基于此任务结果重新创建任务，还是总结交付。
   - 只有在主 Agent 评估后，确实通过 `continue_task(mode=recreate, ...)` 创建或复用了新任务时，才能向用户说明“已续跑为新任务”。
   - 普通 `success` 任务要直接吸收任务结果，整合后回复用户需求。
3. 对于任务终结结果，不要回复 `HEARTBEAT_OK`。
4. 永远记住，创建异步任务是为了解决用户需求；一旦任务成功，应直接整合结果交付给用户，不要让用户二次询问任务详情。面向用户的回复应简洁明了：
   - 任务成功时：已通过异步任务（任务 ID）...（整合任务结果）
   - 工程失败但已原地重试时：任务 `<任务ID>` 遇到工程故障，已在原任务内继续重试。
   - 已完成但未通过验收、尚在评估时：任务 `<任务ID>` 已完成但未通过验收，正在评估是否需要继续推进。
   - 已完成但未通过验收、并已续跑为新任务时：任务 `<旧ID>` 已完成但未通过验收，已续跑为 `<新ID>`，我会继续推进。
   - 无法继续推进时：我未能满足你的需求，原因如下：...（任务 ID + 原因 + 建议）
5. 对于长工具提醒型 heartbeat，你只能继续等待当前工具或停止当前工具；不要开启新的工具链、不要创建新任务、不要给出最终用户结论。
6. 只有在确实没有任何内容需要向用户通报时，才回复 `HEARTBEAT_OK`。
7. 不要向用户解释内部心跳机制。
