# Heartbeat Rules

这些规则适用于心跳驱动的内部轮次。

1. 任务终结结果意味着任务已达到最终状态，如 `success` 或 `failed`。
2. 如果心跳事件是任务终结结果：
   - 遇到 `engine_failure` 时，不要默认创建 continuation 新任务；应优先使用原任务原地重试语义。
   - 如果当前上下文已经显示原任务正在原地重试，只能告诉用户“原任务正在继续推进”，不能伪造“已自动续跑为新任务”。
   - 遇到 `business_unpassed` 时，不要直接假设必须 continuation；应先表述“任务已完成但验收未通过，正在评估是否需要继续推进”。
   - 只有在主 Agent 评估后，确实创建或复用了 `continuation_of_task_id=<旧任务 id>` 的新任务时，才能向用户说明“已续跑为新任务”。
   - 如果评估后不需要 continuation，则直接基于现有结果收尾，不创建新任务。
   - 普通 `success` 任务要直接吸收任务结果，整合后回复用户需求。
3. 对于任务终结结果，不要回复 `HEARTBEAT_OK`。
4. 永远记住，创建异步任务是为了解决用户需求；一旦任务成功，应直接整合结果交付给用户，不要让用户二次询问任务详情。面向用户的回复应简洁明了：
   - 任务成功时：已通过异步任务（任务 ID）...（整合任务结果）
   - 工程失败但已原地重试时：任务 `<任务ID>` 遇到工程故障，已在原任务内继续重试。
   - 已完成但未通过验收、尚在评估时：任务 `<任务ID>` 已完成但未通过验收，正在评估是否需要继续推进。
   - 已完成但未通过验收、并已续跑为新任务时：任务 `<旧ID>` 已完成但未通过验收，已续跑为 `<新ID>`，我会继续推进。
   - 无法继续推进时：我未能满足你的需求，原因如下：...（任务 ID + 原因 + 建议）
5. 只有在确实没有任何内容需要向用户通报时，才回复 `HEARTBEAT_OK`。
6. 不要向用户解释内部心跳机制。
